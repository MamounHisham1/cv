  ---
  1. Prepaid Credits

  Users buy credit packs upfront. Credits are deducted per AI call based on
  actual token cost.

  Plans could look like:

  ┌──────────┬─────────┬───────┬─────────────────────────────────┐
  │   Pack   │ Credits │ Price │            Best for             │
  ├──────────┼─────────┼───────┼─────────────────────────────────┤
  │ Starter  │ 50      │ ~$5   │ Trying it out (~15 evaluations) │
  ├──────────┼─────────┼───────┼─────────────────────────────────┤
  │ Standard │ 200     │ ~$15  │ Regular use (~60 evaluations)   │
  ├──────────┼─────────┼───────┼─────────────────────────────────┤
  │ Pro      │ 500     │ ~$30  │ Heavy use (~150 evaluations)    │
  └──────────┴─────────┴───────┴─────────────────────────────────┘

  How credits are deducted: Each AI call costs credits proportional to tokens
  used. You set a markup (e.g., 2-3x the raw API cost) to cover your margin. For
   example, if an evaluation costs you $0.02 in API tokens, you charge the user
  ~4-6 credits (where 1 credit ≈ $0.01).

  Pros: No surprise bills for users. You get paid upfront. Easy to understand.
  Cons: Users might hesitate to buy before trying. Hard to price credits
  accurately without usage data.

  ---
  2. Post-Pay (Metered Billing)

  Users use the AI freely, then get charged for what they actually used.

  Plans could look like:

  ┌───────────────┬────────────┬────────────────────────┬───────────────────┐
  │     Tier      │  Monthly   │        Includes        │      Overage      │
  │               │    fee     │                        │                   │

  ├───────────────┼────────────┼────────────────────────┼───────────────────┤
  │ Pro           │ $20/month  │ 50 evaluations         │ ~$0.08/evaluation │
  │               │            │ included               │                   │
  ├───────────────┼────────────┼────────────────────────┼───────────────────┤
  │ Enterprise    │ $50/month  │ 200 evaluations        │ ~$0.05/evaluation │
  │               │            │ included               │                   │
  └───────────────┴────────────┴────────────────────────┴───────────────────┘

  How billing works: Track token usage per user. At month-end, charge their card
   based on actual consumption.

  Pros: Users feel less friction — try before paying. Fair since they only pay

  Cons: Requires Stripe Billing with metered usage. Users might get surprise

  ---
  Approach 1: Middleware + Service Class (Recommended)

  A centralized CreditManager service that wraps every AI call. A middleware
  checks credit balance before allowing AI operations. Tokens are deducted after
   each call using the response's Usage object.

  How it works:

  - A ConsumesCredits attribute/middleware on Livewire methods that call AI

  Cons: Slightly more files, needs discipline to route all AI calls through it

  Approach 2: Observer on AI Responses

  Listen for AI response events and automatically deduct credits. Less code but
  harder to control (what if you want to check balance before the call?).

  Pros: Automatic, no manual integration
  Cons: Can't block calls when credits are empty. Harder to customize
  per-operation rules. Observer runs after the expensive API call already
  happened.

  ---
  Approach 3: Decorator Pattern on Agents

  Wrap each AI agent in a CreditTrackingAgent that checks balance before
  forwarding the call and deducts after. Clean but requires modifying every
  place agents are instantiated.

  Pros: Transparent to the calling code
  Cons: Every new Agent() call needs wrapping. Easy to forget. Doesn't work well
   with the builder agent's conversational nature.

  My recommendation: Approach 1 — it gives you control (check before calling),
  flexibility (different rules per operation), and works uniformly across all
  three AI agents. The free builder messages cap is easy to implement as a

  Section 4: Error Handling & Edge Cases

  When credits are insufficient:

  - Show a friendly message: "You're out of credits. Invite friends to earn
  more, or upgrade your plan."
  - Include a link to the referral page
  - The AI call is blocked before it happens — no wasted API cost

  Race conditions:

  - A user could trigger two AI calls simultaneously (e.g., double-click
  evaluate)
  - Use database-level row locking (lockForUpdate() on credit_balances) inside a
   DB transaction
  - If balance goes negative due to race, the transaction fails and the user
  sees an error

  Token usage reporting 0:

  - Some Ollama configurations may return 0 for token counts
  - Fallback: charge a minimum credit cost per operation type (configurable)
  - Log a warning so you know when the fallback is triggered

  Monthly reset edge cases:

  - User registers on Jan 15 → first reset on Feb 15, then Mar 15, etc.
  - If user didn't use last month's credits, they don't stack — balance resets
  to the plan's monthly amount
  - This prevents hoarding

  Referral abuse prevention:

  - One referral code per user, one referral per invitee (can't be referred
  twice)
  - Rate limit registration from same IP
  - Referral code must be used during registration, not after
  - Purchase bonus only triggers once (tracked via purchase_rewarded boolean)

  Builder chat free message tracking:

  - Count messages per conversation (using agent_conversations table that
  already exists in your DB)
  - When free_builder_messages is exceeded, normal credit deduction kicks in
  - Counter resets with each new conversation

  So we can't stop mid-response. What we can do:

  Post-call check (what actually works):
  1. AI call completes, you get $response->usage
  2. Calculate credits needed (e.g., 16)
  3. Compare to balance (e.g., 10)
  4. If credits needed > balance × 1.5 (e.g., 10 × 1.5 = 15, and 16 > 15):
    - Don't deduct anything — return full balance
    - Show error: "This operation required 16 credits but you only have 10.
  Please top up."
    - The AI response is discarded (not saved, not shown)
  5. If credits needed ≤ balance × 1.5:

    - Show the response normally

  - Small overages (within 150%) are forgiven — user gets the response
  - Large overages (>150%) are caught — user keeps credits but loses the
  response

  The trade-off is you eat the API cost when you discard a response. But this
  only happens when a user with very few credits sends something unexpectedly
  expensive.
